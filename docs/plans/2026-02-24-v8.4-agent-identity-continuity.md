# v8.4 Agent Identity Continuity (Engram-Native) Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add an Engram-native identity continuity system that helps agents preserve stable identity signals across resets, compaction, and long-running operation without relying on external project-specific conventions.

**Architecture:** Implement a dedicated identity continuity layer with three parts: (1) recovery anchor documents, (2) continuity incident logging + audit workflows, and (3) budgeted identity injection controls. Keep all features generic, optional, and fail-open. Never modify OpenClaw session files or pointers; this plan is strictly within plugin-safe boundaries.

**Tech Stack:** TypeScript, existing Engram storage/orchestrator/tools/cli modules, markdown+YAML memory store, existing compounding and shared-context subsystems, node:test.

## Design Constraints (Non-Negotiable)

1. **Open-source safe:** No user-specific assumptions, account schemas, or host-specific paths.
2. **Fail-open:** Any identity continuity failure must not break baseline recall/extraction.
3. **Budget-safe:** Identity context injection must be bounded and observable.
4. **No session pointer manipulation:** Plugin must not edit OpenClaw session files/pointers.
5. **Config-gated rollout:** New behavior disabled by default except lightweight diagnostics.

## Functional Requirements

### R1: Recovery Anchor Artifact
- Introduce a structured identity anchor artifact (`identity-anchor.md`) under memory storage.
- Purpose: capture stable identity traits, communication preferences, operating principles, and continuity notes.
- Distinct from normal prompt-time recall; used primarily for recovery/continuity operations.

### R2: Continuity Incident Artifact
- Add append-only continuity incident records with lifecycle states (`open`, `closed`).
- Each incident must capture: trigger time window, observed symptom, suspected cause, applied fix, verification result, preventive rule.
- Records are auditable and immutable except lifecycle-state transitions.

### R3: Continuity Audit Workflow
- Add weekly/monthly continuity audit outputs with a deterministic template.
- Include drift checks (identity changes), stale rule detection, and one next hardening action.
- Integrate with compounding summaries where possible.

### R4: Identity Injection Budgeting
- Add explicit char/token budget controls for identity-context injection.
- Add mode controls:
  - `recovery_only` (default)
  - `minimal`
  - `full`
- Ensure identity context never crowds out core recall sections under constrained budgets.

### R5: Improvement Loop Register
- Add a generic recurring-improvements register artifact (cadence, purpose, status, kill condition, last reviewed).
- Enable agent+operator visibility into which continuity loops are active vs stale.

### R6: Human-Confirm Safety Hooks
- For any operation classified as high-risk continuity action, expose recommendation + checklist only.
- Do not execute destructive or session-wiring actions automatically.

## Non-Goals

1. No direct backup transport implementation to cloud providers in this slice.
2. No OpenClaw core changes (separate blocked plan already exists).
3. No automatic personality rewrites based on transient short-term signals.
