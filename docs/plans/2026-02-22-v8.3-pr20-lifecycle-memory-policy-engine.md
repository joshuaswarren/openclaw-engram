# v8.3 PR20: Lifecycle Memory Policy Engine (MemoryOS-Inspired, Engram-Native)

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

## Objective

Implement a deterministic, policy-driven lifecycle engine that governs **promotion, decay, revalidation, and retrieval eligibility** for memories using existing Engram signals (access, recency, confidence, feedback, contradiction status).

This adds the most transferable MemoryOS primitive (lifecycle promotion logic) without replacing Engram’s current architecture.

## Rationale (linked to BM-493923)

BM-493923 found that MemoryOS’s key advantage is not storage format but a coherent lifecycle loop (short→mid→long with heat-triggered promotion). Engram already has stronger operations/safety and already ships:

- memory statuses (`active/superseded/archived`)
- access tracking (`accessCount`, `lastAccessed`)
- confidence tiers + TTL
- consolidation/archival passes
- feedback and negative-example biasing

Current gap: no unified lifecycle score/state machine spanning these signals. Lifecycle decisions are split across ad-hoc rules (speculative expiry, commitment decay, low-importance archival). PR20 closes that gap with an explicit policy layer.

## Architecture Changes

### 1) New lifecycle dimensions

Retain existing `status` (`active/superseded/archived`) and add policy-level dimensions:

- `lifecycleState`: `candidate | validated | active | stale | archived`
- `verificationState`: `unverified | user_confirmed | system_inferred | disputed`
- `policyClass`: `ephemeral | durable | protected`
- `lastValidatedAt?: string`
- `decayScore?: number` (0-1)
- `heatScore?: number` (0-1)

Notes:
- `status` remains storage/index state.
- `lifecycleState` is retrieval/promotion semantics.
- Mapping rule: if `status=archived`, lifecycle must be `archived`.

### 2) Lifecycle scoring engine

Add `src/lifecycle.ts`:

- `computeHeat(memory, now, signals)`
- `computeDecay(memory, now, signals)`
- `decideLifecycleTransition(memory, policy, now)`

Signals consumed (already available in Engram):
- recency (`created`, `updated`, `lastAccessed`)
- usage (`accessCount`)
- confidence (`confidence`, `confidenceTier`)
- feedback (`relevance`, negative examples)
- contradiction/supersession markers
- importance score

### 3) Consolidation integration

In `orchestrator.ts` consolidation flow:

1. load active memories
2. run lifecycle score pass
3. apply transitions (`candidate→validated→active`, `active→stale`, `stale→archived`)
4. persist changed frontmatter (only changed records)
5. emit lifecycle metrics snapshot (`memory/state/lifecycle-metrics.json`)

### 4) Retrieval integration

In retrieval post-processing (`boostSearchResults` + pre-filter path):

- optional filter of `stale` memories (`lifecycleFilterStaleEnabled`)
- additive lifecycle boost/penalty:
  - `validated/active` small boost
  - `candidate` neutral/slight penalty
  - `stale` penalty (or filtered)
  - `disputed` stronger penalty

Fail-open: if lifecycle metadata missing, default to current behavior.

### 5) Config additions

Add to `PluginConfig`, `config.ts`, and `openclaw.plugin.json`:

- `lifecyclePolicyEnabled` (default `false`)
- `lifecycleFilterStaleEnabled` (default `false`)
- `lifecyclePromoteHeatThreshold` (default `0.55`)
- `lifecycleStaleDecayThreshold` (default `0.65`)
- `lifecycleArchiveDecayThreshold` (default `0.85`)
- `lifecycleProtectedCategories` (default: `decision,principle,commitment,preference`)
- `lifecycleMetricsEnabled` (default `true` when policy enabled)

## Migration / Compatibility Notes

- Backward compatible: all lifecycle fields optional.
- Legacy memories load as current behavior; inferred defaults only in-memory unless changed.
- No forced reindex required; optional `qmd update` recommended after large transition batches.
- Existing `status` semantics are preserved.
- If `lifecyclePolicyEnabled=false`, behavior remains unchanged.

---

## Risks and Mitigations

1. **Over-demotion of useful memories**
   - Mitigation: start with ranking penalties before hard filtering; protected categories list.

2. **Frontmatter churn causing disk overhead**
   - Mitigation: write only when transition/score delta exceeds threshold.

3. **Policy brittleness across workloads**
   - Mitigation: explicit config thresholds + metrics-driven tuning.

4. **User surprise from hidden retrieval changes**
   - Mitigation: add debug visibility in metrics and optional lifecycle explain output.

5. **Interaction with existing archival logic**
   - Mitigation: lifecycle pass runs before archival; archival remains final gate.
